What OINP is

OINP is the company-wide platform every Intuit product team (QuickBooks, TurboTax, Credit Karma, Mailchimp, Field Service, IEP, ProSeries, Lacerte, Acadia) uses to deliver transactional and marketing notifications across Email, SMS, RCS, Voice, Push, Tray, In-App, and ICE. The platform spans authoring (RFM), delivery (channel adapters), event ingestion (OMS/Pulsar), and the management surfaces (search, dashboards, promotion).

I own the engineering for the modernised authoring stack, the platform's tax-peak reliability program, and the cross-org integration surface. Concretely: five backend services (comms-authoring-service, authoring-delivery-sync, comms-comparator, comms-promotion-service, the legacy notificationadmin being retired), two channel adapters (smsadapter, oinp-tray), the OINP onboarding documentation that partner teams consume, and the game-day / HPA / scaling program that gates each tax peak.

Directional decisions across the work

  • Migrate, don't patch, when patch budget exceeds migration budget. Made this call publicly in #customer-comms-pd for the Notification Tray (40 GB Cassandra+Solr → managed DB), drove it for the legacy authoring stack (RFM Revamp), and pushed for elimination of the legacy scheduling service before TY25. Each removed a chronic failure source.
  • Parity over hope. Stood up comms-comparator as the parity backbone for the entire RFM migration. Ran a 7-day prod audit: 2.9M reads, 99.987% parity, every failing class root-caused. Without it, we could not have moved 100% of traffic to the new stack before tax peak.
  • Phased rollout to avoid throwaway work. S3 thumbnails → Elasticsearch → domain-event resiliency, each phase usable in its own right. Default approach across the team now for "add capability X to product Y."
  • API contract before implementation. Pushed this with every junior engineer touching a new surface. Pushed back in #customer-comms-pd-leaders against a peer's BFF proposal — used notif-common's history of scope-creep as the precedent, and proposed gateway routing as the simpler primitive. The decision held; no BFF.
  • Escalate to leadership when peer-level chatops stalls. When the OMS team's customer-facing channel wouldn't approve cross-swim-lane routing to prod (blocking VEP and Field Service integration), escalated in leaders channel for manager contacts, scheduled the cross-team meeting myself, negotiated a prod rollout timeline.
  • Demonstrate before proposing. Rather than write a doc for the home-page redesign, generated previews of 3,550 production notifications using the new Preview API and hosted the visual mockup. The proposal landed because the data was already there.
