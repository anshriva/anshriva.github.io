RFM Search — two ships, two data sources

Topic · Intuit · OINP / Customer Communications

Problem statement

The notification authoring portal (RFM) holds thousands of notifications across hundreds of workspaces and tens of product teams. To find "the notification that fires on QBO invoice payment" an author had to know its catalog, scroll its event list, or ask the platform team in Slack. There was no global search across notifications, message templates, fragments, and the metadata that ties them together.

I have shipped search for RFM twice, on two completely different stacks with different data sources:

  • Ship 1 — Sep 2024. Search on the legacy stack. DynamoDB source → Kinesis stream → IPS Search index. Shipped, operational, still serving production traffic on legacy RFM.
  • Ship 2 — 2026 (in flight, backend nearly complete by GED-end). A new build for the new RFM stack. IPS Managed Relational (Aurora Postgres) source — a different data substrate that needs a different ingestion model and a different schema. This is not a refactor of Ship 1; it's a new system.

Context & constraints

  • Two separate ships, not two phases. Ship 1 was a complete, operational system on the legacy notificationadmin stack. Ship 2 is being built fresh on the new comms-authoring-service stack with a different source-of-truth database. The product surface looks similar to authors; under the hood almost nothing carries over.
  • The data source changed. Ship 1's source was DynamoDB — change-data via Kinesis. Ship 2's source is IPS Managed Relational (Postgres). Different CDC story, different schema, different join semantics.
  • Data freshness still matters. Authors edit a notification and immediately want to find it. Batch reindexing would have been simpler in either ship but unusable.
  • Sort and ranking expectations were unbounded at first. Product asked for "sort by every field" — I pushed back with a curated whitelist driven by what authors actually needed and what an index can support without performance collapse.
  • Phased rollout strategy for Ship 2. Rather than build everything before shipping, I structured Ship 2 in three deliberately phased deliveries so each phase is usable on its own and the next phase builds on it.

Ship 1 — legacy RFM search (Sep 2024)

Built on the legacy notificationadmin stack, backed by IPS Search (Intuit's OpenSearch offering) for the index and DynamoDB + Kinesis as the change-data pipeline. Shipped, fully operational, still serving prod.

  • Any change in our DynamoDB tables emitted to a Kinesis stream.
  • A backend consumer combined data from multiple tables into a single JSON document and pushed it to the IPS Search ingestion API.
  • The UI called a search endpoint that translated user query + filters into IPS Search queries, returning ranked results.
  • I designed it event-based (not batch) because manual data freshness guarantees don't survive prod — relying on the stream gave us "always within seconds of write" without operator effort.

For ranking I capped the sortable field set rather than supporting every requested field: name, offeringName, serviceName, modifiedByMail, modifiedAt, createdByEmail, createdAt, storageExpiry, storageDeletion — plus relevance. Pushed back on sorting by email body / notification description because each sortable field has a measurable performance cost on every query.

I also added a new gateway route for /genai with a higher timeout (5s wasn't enough for the gen-AI integration), tested it in QAL, verified the RFM UI worked end-to-end, then promoted the gateway config to e2e and prod.

Ship 2 — new RFM search (2026, in flight)

For the new RFM stack the source-of-truth is IPS Managed Relational (Aurora Postgres), not DynamoDB. The architectural primitives change accordingly — different CDC pattern, different schema-mapping, different joining model — so this is a fresh build, not a port. I structured it in three deliberately phased ships so each is independently useful:

  • Phase 1 — Thumbnails in S3, no Elasticsearch. The smallest amount of indexing infrastructure needed to make the new RFM searchable. If something went wrong, we caught it via Splunk alerts and retried manually. Faster to ship, no event-bus dependency. Backend nearly complete; on track for prod by GED end (Jun 2026).
  • Phase 2 — Elasticsearch. Once Phase 1 is in use we layer Elasticsearch underneath for real ranking + facets, swapping in without breaking the API contract Phase 1 established.
  • Phase 3 — Domain-event resiliency. Replace manual retry with domain-event-driven ingestion so the index stays converged with the source of truth even under partial failure. This is the long-term operational story.

I mentored a junior engineer (Suveeksha Reddy) on owning Phase 1 backend, then a second engineer (Himaja) on the Phase 2 search API design — pushing both to think in terms of API contracts first, and to refer to the existing notificationadmin search contract (i.e. Ship 1, which I'd built) instead of exposing every Elasticsearch construct (e.g. the filter array, the AND-only operator pattern that's enough for OINP without the composite NOT/OR complexity).

Ship 2 is also the backbone for the home-page preview redesign — see OINP Accelerate for how the search index and the Preview API combine into a visual-first home page.

Key decisions & tradeoffs

  • Build fresh for the new stack, don't port. Once I knew the source-of-truth was Aurora Postgres rather than DynamoDB, the right move was a clean build rather than wrapping the old ingestion pipeline. Ship 1's stream-based DynamoDB ingestion doesn't translate; the relational source has different CDC primitives, different schema constraints, different join opportunities. Trying to make one codebase serve both stacks would have left us with the worst of each. Tradeoff: two systems to maintain during the migration window; Ship 1 retires when legacy RFM retires.
  • Event-driven ingestion in both ships, batch in neither. Search staleness is the kind of bug authors notice immediately ("I just saved it and it's not in search yet"). Different implementations (DynamoDB Streams + Kinesis for Ship 1; domain events for Ship 2 Phase 3) — same principle: writes propagate to the index in seconds, no operator effort. Tradeoff: more failure modes than batch; mitigated by Splunk dashboards and saved searches.
  • Curated sort fields, not "sort by everything." Product asked for sort on the full schema. I refused for fields that don't actually help authors (email body, notification description) and would have made every query slower for everyone. Tradeoff: a few authors didn't get exactly what they asked for; the system stayed fast for everyone.
  • Ship 2 Phase 1 = no Elasticsearch. The cleaner long-term answer is Elasticsearch from day one. But Phase 1 with S3 thumbnails is something one engineer (Suveeksha) can ship in a sprint and prove the API surface against real users. Phase 2 swaps in ES without changing the API. Tradeoff: throw-away resiliency in Phase 1 (manual Splunk retries); the whole point is that it gets replaced before scale matters.
  • AND-only operators in the Phase 2 filter API, not full boolean. When Himaja was designing the Phase 2 search API, she wanted to expose Elasticsearch's full operator set (AND / OR / NOT, composite). I pushed back: OINP's actual query patterns are AND-only, and exposing composite operators leaks ES specifics into our API. We can add later if needed. Tradeoff: less expressive; matches what authors actually do, and matches the Ship 1 contract.
  • Gateway timeout extension for /genai (Ship 1, 5s → higher). The gen-AI integration needed more than the platform default. Rather than fighting upstream limits or removing the gen-AI path, I added a dedicated route with the higher timeout, tested in QAL, then promoted. Tradeoff: a path-specific carve-out — but cheaper than re-architecting the integration.

Impact

  • Ship 1 — shipped Sep 2024 as the search experience in legacy RFM. Still serving production. Initial usage ~40–50 searches/day, fluctuating, growing as authors discovered it and bookmarked the dashboard.
  • Ship 2 — backend nearly complete by GED-end (Jun 2026) as the search experience for the new RFM stack. Designed to look "same as intermediate RFM but much better" once integrated with the visual home-page redesign.
  • Mentorship outcome: Suveeksha owns Phase 1 backend; Himaja owns the Phase 2 search API design. The "look at the existing notificationadmin contract before exposing new constructs" rule is now how API reviews go on the team — established during the Himaja review.
  • Pattern reuse: the 3-phase rollout pattern (build minimal → add backbone → add resiliency) is now the default approach the team uses for "we want to add X to Y" asks.

FAQ (for future recruiter agent)

Why didn't you reuse Ship 1's code for Ship 2?
The source data is in a different system (DynamoDB vs. Aurora Postgres), with different CDC primitives, different schema constraints, and different join opportunities. The right abstraction layer to share between them would be the search API contract — which we do share, because I pushed Phase 2 to match the Ship 1 contract intentionally. But the ingestion + indexing code is necessarily new.
How do you handle joins across multiple tables in the index?
Ship 1 — the consumer code combined data from multiple DynamoDB tables into a single denormalised JSON before ingesting. Ship 2 — same pattern at the storage layer (denormalised search docs because that's what makes search fast), but the source has real joins available now, so building the denormalised doc is cleaner. In both cases the tradeoff is "doc gets stale if you forget a write path" — mitigated by event-driven ingestion that fires on any source mutation.
Why IPS Search instead of running your own OpenSearch cluster?
IPS Search gives us the paved-road benefits: integrated auth, password complexity, SSL, regional DR, promotion via standard env progression, and 1-hour SLA on schema changes. Running our own cluster would give us more flexibility with no team-wide benefit. We took the paved road for both ships.
What happens when the search index falls behind the source of truth?
Ship 1 — Splunk alert fires, we manually re-ingest the lagging documents. Ship 2 Phase 1 inherits that pattern temporarily. Ship 2 Phase 3 (the resiliency phase) replaces this with domain-event-driven retries so convergence is automatic. The transition is deliberate — early phases are "is this useful enough to invest more"; Phase 3 is the answer.
What's the migration plan when legacy RFM retires?
Ship 1 retires with the legacy stack. Search traffic moves to Ship 2 as workspaces migrate. No data migration of the search index itself is needed — the source-of-truth migration (notification data Aurora → Aurora) is the work; the search index just rebuilds against whichever source is authoritative.
