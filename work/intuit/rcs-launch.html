RCS channel — end-to-end ownership of a brand-new bearer

Topic · Intuit · OINP / Customer Communications

Problem statement

RCS (Rich Communication Services) is the carrier-native successor to SMS — rich cards, suggested replies, media, two-way conversations, all on the user's default messaging app with no install required. For Intuit, RCS unlocks experiences SMS can't: Field Service confirmations with quick-reply buttons, Critical Money Alerts with embedded actions, Triton conversations with structured media. It's also a channel none of OINP had built before.

I owned the RCS channel end-to-end on OINP: vendor integration (Twilio), schema and API design on the new authoring stack, the two-way (inbound) message path, the cross-swim-lane routing to consumer teams, consent and SMS-fallback semantics, the canonical onboarding docs the rest of the org uses, and the working-group leadership that aligned three internal teams (Triton / VEP, Field Service, Credit Karma) on a single delivery path.

Context & constraints

  • No prior RCS at Intuit on OINP. Zero in-house schema, no adapter precedent, no debugability playbook — everything from "how does an author configure an RCS template in RFM" to "how do we know if an inbound reply was received" had to be defined.
  • Twilio's new RCS API was in preview. The richer comms.twilio.com/preview/Messages API (priority-based bearer selection, structured rich-content modules, native SMS fallback in one call) was emerging while we were building. I had to choose between shipping on the stable Content Templates path now vs. waiting for the new API.
  • Two-way RCS is structurally different. Inbound RCS replies arrive via Twilio's SMS-shaped webhook (SmsMessageSid, MessagingServiceSid) but carry the RCS payload inside a ChannelMetadata JSON blob with provider-specific fields (Google's RBM sender id, googleapis blob URLs with 4-hour expiry alongside Twilio CDN URLs). Parsing this correctly is non-obvious from the docs.
  • Multiple consumer teams, different swim-lanes. Triton (VEP) and Field Service both wanted RCS replies routed to their topics — but they sit in different OMS swim-lanes from OINP. Cross-swim-lane routing is a 7-day-of-work ask from the OMS team and they initially resisted enabling it in prod.
  • SMS fallback expected, not optional. Not every recipient phone is RCS-eligible (carrier, device, region). If RCS-ineligible users get nothing, the feature is unusable. Fallback has to happen at delivery time, transparently to the author.

What I built end-to-end

1. Authoring schema on the new RFM stack

Designed the RCS Communication payload on comms-authoring-service. Key shape:

  • channel: "RCS" with a messageTemplate carrying contentSid (Twilio Content Template id), contentVariables with FreeMarker substitutions resolved from event payload (${jsonEventData.name}), and rcsId binding to the configured Twilio sender.
  • deliveryConfiguration with toAddress: "rcs:${jsonEventData.phone}" (the rcs: prefix is how Twilio identifies the bearer), a vendorConfig per workspace, and the two RCS-specific switches: applicableConsentCheck (EXPLICIT_CONSENT_CHECK vs IMPLICIT_CONSENT_CHECK depending on use case) and allowSmsBackup: true as the SMS-fallback opt-in.

2. Twilio integration — outbound

Shipped first against Twilio's stable Content Templates path — author creates the template in Twilio console, we reference it by contentSid and inject variables at send time. This is what's in production now. I explicitly chose this over waiting for Twilio's new preview API because the new API was gated and I didn't want to block first-customer launch on Twilio's release cadence.

In parallel I evaluated the new comms.twilio.com/preview/Messages rich-content API for the next phase — single-call multi-bearer priority (RCS first, SMS fallback) with structured modules (rich cards, standalone cards, suggestions with openUrlAction). Flagged early to leaders that we should adopt the new API before the first customer onboards, because otherwise existing templates would need migration later: "backend wise i see max 2 days (or even less) to make change. I think we should make this change before first customer onboards else there will be hassle to migrate the existing ones to new api."

3. Two-way RCS — inbound webhook

Twilio delivers inbound RCS via the same webhook shape as SMS, but the RCS payload lives inside a ChannelMetadata JSON blob: 

{
  "type": "rcs",
  "data": { "context": { "channelPayload": {
    "senderPhoneNumber": "+919538512471",
    "messageId": "Mxus9eLogOQdmAPTikVw4x4A",
    "sendTime": "2026-04-24T04:38:03.105134Z",
    "agentId": "test_oinp_zmxsa4tu_agent@rbm.goog",
    "text": "Okay",
    "provider": "google"
  } } }
}

I drove the parsing logic in textmsginboundsvc, set up the webhook against the RCS sender in the Twilio console, and validated end-to-end including media uploads (a separate path where inbound carries NumMedia, MediaUrl0, and MediaContentType0). Caught and escalated a Twilio doc gap to Govind / Michael at Twilio: the inbound media URLs (one from rcs-copper-ap.googleapis.com, one from mms.twiliocdn.com) don't require authentication despite the docs claiming they should, and have different expiry semantics — important for any downstream service that re-serves the media.

4. Cross-swim-lane OMS routing — two-way replies to consumer teams

Inbound RCS replies have to reach the team that owns the conversation — Triton (in the VEP swim-lane) for general two-way RCS, the Sales Feeder Service (in the SBSEG swim-lane) for specific flows. Designed and drove the OMS multicast routing setup:

  • Source topic (OINP swim-lane): persistent://oinp/regcomm/public.user.reply
  • Destination 1 (VEP swim-lane): persistent://vep/collaboration/public.oinp.smsRcsEvent7216
  • Destination 2 (SBSEG swim-lane): persistent://sbseg/payments/public.salesfeederservice
  • Filter: header-based, e.g. to == 'rcs:test_oinp_zmxsa4tu_agent' — each consumer team gets only the replies addressed to their sender.

When the OMS team approved this routing in e2e but resisted enabling it in prod (citing 7 days of work, and stopping our new e2e routing in the process — blocking VEP integration entirely), I escalated to leaders channel asking for OMS manager-level contacts, scheduled the cross-team meeting with Abhinesh / Anaghesh / Gaurav from OMS, and unblocked both e2e and a prod timeline within a week.

5. SMS fallback when RCS is ineligible

Two pieces had to be in place. Content fallback: every Twilio Content Template carries a Text fallback body alongside the rich content; when the recipient isn't RCS-eligible, Twilio sends the Text body via the SMS backup sender. Infrastructure: fallback only works if a backup short code / long code / toll-free number is wired on the same Twilio Messaging Service as the RCS Sender. If those aren't wired together, RCS-ineligible users get nothing. I documented both gotchas in the onboarding guide because the failure mode is silent.

6. Consent capture

RCS in the US uses an applicationId + consentType + purpose tuple from the consent platform. I drove the cross-team request to #iip-consent-help for a new consent application id covering SMS + RCS for the Critical Money Alerts use case. Keyword-based consent capture (STOP / UNSUBSCRIBE / CANCEL → opt-out; START → opt-in) is configured per workspace per channel in the SMS adapter config — the same config primitive serves both bearers, with the shortOrLongCode being either an SMS phone number (+18669553710) or an RCS sender (rcs:test_oinp_zmxsa4tu_agent).

7. The canonical onboarding docs

Wrote the full RCS section of the OINP onboarding guide (PR OINP-21070, doc PR docs/pull/107) — 8 pages covering:

  • Prerequisites
  • Setup walkthrough
  • Testing the flow
  • Twilio templating reference
  • Incoming messages (two-way RCS)
  • Consent capture
  • Debugging RCS delivery
  • Going live (e2e → prod)

Then took it one step further: shipped an rcs-docs skill (PR oinp-skills/pull/59, OINP-21322) that lets engineers query the RCS docs via Claude Code locally without browsing — answers RCS questions from the canonical doc set so the docs aren't just static reference but a queryable knowledge base.

Key decisions & tradeoffs

  • Ship on the stable Content Templates path first; new preview API as Phase 2. Waiting for Twilio's new comms.twilio.com/preview/Messages API would have delayed first-customer launch on RCS by an undefined window. Shipping on Content Templates now means first customers can onboard immediately. I also flagged the migration cost early to leaders so we'd adopt the new API before first customer onboards (≤2 days of backend change) rather than retrofitting customer templates later. Tradeoff: a small window where templates live in Twilio console rather than authored entirely in RFM; closed before first prod customer.
  • Reuse SMS adapter for RCS rather than build a new rcsadapter. RCS and SMS share the Twilio Messaging Service, the consent semantics, the keyword-action mappings, and the inbound webhook shape. A separate rcsadapter would have doubled the operational surface for ~0 net benefit. Extended the existing SMS adapter config (the shortOrLongCode field already accepts the rcs:-prefixed sender id) — same code path, different bearer. Tradeoff: one service handles both channels; mitigated by per-channel config and clear logging.
  • Header-based OMS routing, not application-level filtering. Could have routed all inbound replies to every consumer team and let each filter their own. Header-based filtering at the broker means each team only ever sees the messages addressed to their sender — less code in each consumer, no risk of a consumer accidentally processing another team's replies. Tradeoff: more upfront work with the OMS team; pays off as we add more consumer teams.
  • Escalate, don't wait, when the OMS team blocks. The OMS approval for cross-swim-lane prod routing was stalled in a customer-facing chatops channel. I escalated to platform leaders for OMS manager-level contacts and scheduled the cross-team meeting myself rather than waiting. Tradeoff: spends political capital; the alternative was VEP and Field Service blocked indefinitely, which was worse.
  • Permissive parsing on inbound RCS payloads. The vendor's webhook payload has fields that vary by message type (with-media vs text-only), provider (Google RBM today; Apple in future), and Twilio API version. Strict schema validation would have rejected legitimate inbound messages. Defensive parsing — extract what's there, log anomalies, don't fail. (Same principle I applied to SMS phone-number parsing; see the SMS launch page.) Tradeoff: less clean validation; matches what the vendor actually sends.
  • Authored the canonical docs + the Claude Code skill, not just code. A new channel that only the author understands isn't really shipped. The 8-section onboarding guide turns RCS into a self-serve capability for partner teams. The rcs-docs skill makes those docs queryable from any engineer's local Claude Code. Tradeoff: ~1 week of writing on top of the code work; the alternative is being the bottleneck for every partner team's RCS onboarding indefinitely.

Impact

  • RCS shipped as a first-class channel on OINP — authoring, delivery, two-way, SMS-fallback, consent, cross-swim-lane routing to consumer teams. Both Triton/VEP and Sales Feeder Service consuming inbound RCS replies via the OMS routing.
  • Field Service / VEP / Credit Karma integrations unblocked. The two-way path (user replies routed to the consumer team's topic) is what these teams needed. Without the cross-swim-lane OMS routing, none of them could have launched.
  • Critical Money Alerts (Apr 2026 Salute award) uses the SMS+RCS infrastructure — the Critical Money Alerts experiment is one of the first production consumers of this work.
  • OINP onboarding guide for RCS is the canonical doc on the platform docs site — every partner team that onboards RCS uses it; PR OINP-21070 brought it in, PR OINP-21322 made it queryable via the rcs-docs Claude Code skill.
  • Volunteered in leaders channel to lead through prod: "Will lead this till production if that is ok by GED end it should be live."
  • Pattern reuse: the SMS-adapter-handles-RCS pattern, the header-based OMS routing pattern, and the docs-as-skill pattern have all become defaults for the next bearer onboarding.

FAQ (for future recruiter agent)

Why not adopt Twilio's new preview API from day one?
Access was gated and the API was still moving. Shipping Phase 1 on Content Templates meant first customers could onboard without waiting on Twilio. I flagged the migration cost early to leaders (≤2 days backend) so we'd switch before first prod customer rather than retrofitting customer templates later.
How do you debug a missing RCS delivery?
The debugging guide walks through it: check Twilio console for the message status, check the OINP send logs for the outbound webhook to Twilio, check the consent state (RCS without explicit consent doesn't send), and if the user is RCS-ineligible verify the SMS fallback is wired on the same Messaging Service as the RCS sender.
What about Apple's RCS rollout — won't that change things?
The bearer layer is abstracted behind Twilio's provider field — current inbound payloads carry "provider": "google" (Google RBM). When Apple Messages-for-Business comes online, Twilio handles the bearer; the OINP layer needs zero code changes as long as the Channel Metadata shape stays as documented.
What's the next milestone?
Group conversations (currently exploring Twilio Conversations for native group support — already filed the question with Twilio), and richer RCS card types (carousels with multiple cards, structured suggestions tied to backend actions via postbackData).
