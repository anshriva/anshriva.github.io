OINP Accelerate — proposing a UX redesign by shipping the evidence first

Topic · Intuit · OINP / Unsolicited initiative

Problem statement

The new RFM home page is functional but information-dense — a table of notifications with names, statuses, and modification dates. To understand what a notification actually looks like, an author has to click into it, preview it, and click back. For thousands of notifications across hundreds of workspaces, this friction makes RFM feel like a database browser rather than a creative tool.

I proposed redesigning the home page around a visual-first model — show the notification's actual rendered preview as the primary unit on the home page, with metadata as decoration. This wasn't a roadmap item; it was an initiative I picked up unsolicited because the new Preview API made it suddenly cheap. I shipped working evidence at production scale (3,550 notification previews) before writing the proposal — and then made the case in #customer-comms-pd-leaders.

Context & constraints

  • The Preview API was newly available. Part of the RFM Revamp work — preview generation lived behind /v1/notificationdef/preview/internal and the equivalent surfaces on the new authoring stack. Until this existed, doing visual rendering at scale wasn't possible without per-template manual work.
  • 3,550 notifications in prod-equivalent e2e. Real production notifications, not synthetic test data — meaning the proposal would be evaluated against actual variety: every channel, every catalog, every template complexity.
  • Skeptical audience. Adding a visual layer to every home-page entry could mean 3 API calls per preview × N visible notifications. The "this will be slow / expensive" objection had to be answered before the meeting, not during.
  • Leadership is impressed by working artifacts, not pitches. A doc proposing this would compete with five other docs. A drive full of rendered previews + a working mockup short-circuits the discussion.

How I shipped the evidence

  • Generated 3,550 previews end-to-end using the new Preview API. Ran the API against e2e for every active notification, captured the rendered output, dumped all images to a shared Google Drive. A few failed for legit reasons (asset onboarding for report datasets, mis-configured event data); the majority worked clean.
  • Built a mockup of the redesigned home page hosted on GitHub Pages at github.intuit.com/pages/ashrivastav6/oinp-accelerate/mockups/notification-preview-redesign.html — the visual UI authors would actually see, not a Figma drawing.
  • Pre-answered the latency objection. I worked out two implementation options: (a) build fresh on every page-load — quick to ship but each preview triggers 3 calls to comms-comparator, so the experience would be slow on dense pages; (b) pre-build previews on domain-communication-events and store in IPS Search — same pattern we already proved on legacy notif-admin two years ago. The doc I shared in leaders channel laid this tradeoff out explicitly.
  • Anchored the IPS Search proposal in existing operational data. "It just requires onboarding to IPS Search and we can support preview also. We did that analysis 2 years back for notif-admin and it has been working very reliably." Linked to the original design doc.

Key decisions & tradeoffs

  • Ship working evidence before proposing. The cheap path is to write a doc, attach screenshots, and ask for review. The hard path is to actually run the API at scale, host the output, and build the mockup. The hard path makes the meeting decision-mode instead of debate-mode. Tradeoff: 1–2 weeks of upfront work that could be "wasted" if the idea got rejected; the proposal landed on first review precisely because it didn't feel speculative.
  • Pre-build previews + IPS Search, not on-demand rendering. The simplest implementation (call comms-comparator three times on each preview load) would have been functional but slow. Recommending pre-build + index lookups makes the home page feel instant and reuses infrastructure we already operate. Tradeoff: more moving parts (need to keep the index converged with the source of truth via domain events); same pattern we've proved on legacy notif-admin for 2 years.
  • Use my own GitHub Pages site for the mockup. Could have shared a Figma file or a Google Slides deck. GitHub Pages is faster to iterate on (push, refresh), works for both engineers and product reviewers, and signals "this is engineering-real, not design-speculative." Tradeoff: mockup looks less polished than Figma; conveys intent better.
  • Estimate the work explicitly: "minor work, say 1 week / max 2 weeks." Including the timebox in the original proposal removed the "how big is this" friction. The home page can be much more interactive within a sprint, not a quarter. Tradeoff: a public commitment that has to be hit; the estimate is grounded in the pre-build path already existing as a pattern.

Impact

  • 3,550 production-equivalent notifications rendered through the new Preview API — first time this had been done at scale and proof the API holds up against real-world variety.
  • Mockup hosted at github.intuit.com/pages/ashrivastav6/oinp-accelerate/ — referenced from #customer-comms-pd-leaders and #customer-comms-pd as the concrete artifact for the proposal.
  • Picked up as part of GED 2026 backend search work — backend approximately complete by end of GED week; visual home page is the user-visible payoff of the search + preview infrastructure.
  • Established a pattern within the team. "Ship the evidence, then propose" — the GED 2026 LangGraph agent and the RCS go-live initiative both followed this same working-artifact-first pattern.

FAQ (for future recruiter agent)

What if leadership had rejected the proposal?
The Preview API generation work would still have been useful — it stress-tested the new API against the long tail of production notifications and surfaced real onboarding bugs (report dataset assets, mis-configured event data). And the mockup is on my personal GitHub Pages site, not the org's, so it's recoverable without political cost.
Why is the mockup site called "oinp-accelerate"?
It's a personal name I picked to gesture at "things that would make OINP feel faster to use" — both for authors (visual home page) and for engineers integrating with us (better onboarding docs). The visual home page is the first artifact under that umbrella; the RCS onboarding guide I wrote is another.
How does this connect to the search work?
Tightly. The visual home page only works if you can find notifications quickly — otherwise it's a slow grid. The GED 2026 search work (Elasticsearch-backed, mentored by me, executed by Suveeksha and Himaja) is the prerequisite. We're shipping them as one product surface.
What's the lesson you'd apply to the next initiative?
Working evidence beats pitch decks by a wide margin. The next time I want to change how authors interact with a 1000-notification list, I'll spend a week generating the visual artifact and skip the doc round entirely.
