GenAI authoring — AI-assisted notification creation

Topic · Intuit · OINP / GED 2025

Problem statement

Authoring a new notification end-to-end on OINP — picking the right event, writing the trigger condition, drafting the template per channel, configuring throttling, localising, promoting to prod — historically took up to a week of back-and-forth between a product team and the platform team. Most of that time was friction, not actual writing: finding the right docs, asking the platform team for examples, discovering missing fields after submission.

At GED (Global Engineering Days) in Nov 2025, I led a 48-hour build of an AI-assisted notification authoring agent that turns a plain-English description into a draft notification with templates, triggers, and channel config pre-filled. The team won the Lightning Award for Fastest GED POC, and the agent is now running in production as ccx-oinp-guide-agent, used inside the new RFM portal.

Context & constraints

  • 48-hour build window. GED runs a single week (Nov 17–21 2025). The "Lightning" recognition was for putting out the first end-to-end demo within 48 hours of GED starting.
  • Existing UX norms were the enemy. The new RFM is a form-first UI; layering AI on top of forms is awkward. The bet was that authors would prefer "describe what you want and watch it appear" to "fill in 14 fields."
  • Has to land inside the new RFM, not as a standalone tool. An external "ChatGPT for notifications" wouldn't get adopted; it had to live where authors already worked.
  • Multi-agent architecture, not a single prompt. A single LLM call to "make me a notification" doesn't compose — different aspects (template content, trigger logic, channel config) have different correctness criteria and need different tooling.

Architecture

The agent is a LangGraph-based multi-agent system rooted at a coordinator that decomposes an authoring intent into sub-tasks and routes each to a specialised agent. The repo is ccx-oinp-guide-agent (registered in DevPortal as Intuit.custeng.custcomms.genaianubhavged2025).

  • Coordinator agent. Reads the user's intent, decides which sub-agents to invoke, and stitches the outputs into a coherent draft. The coordinator's system prompt is the load-bearing part — it encodes the OINP authoring model (catalogs → events → notifications → templates) so downstream agents don't have to.
  • Template-drafting agent. Generates FreeMarker / MVEL templates for the requested channels, with locale handling and trigger-context substitution.
  • Trigger / condition agent. Translates user intent into the conditional logic that gates the notification (if-else over event payload).
  • Channel-config agent. Fills in the per-channel config (throttle, storage, vendor selection).

How it landed in prod

The GED demo was the seed; the production version is now ccx-oinp-guide-agent, listed in DevPortal as a genuine GenAI application used in the new RFM (alongside two earlier temp resources that were archived). It survived the GED → prod transition because the architecture wasn't "demo glue" — it was a coordinator + tools pattern that mapped cleanly onto how authors actually work.

Key decisions & tradeoffs

  • Multi-agent with a coordinator, not a single fat prompt. A single prompt would have been faster to build in 48 hours but wouldn't have scaled — each subdomain (templates, triggers, channels) has its own correctness model. Coordinator + specialists let each agent be evaluated independently and swapped without breaking the others. Tradeoff: more orchestration complexity; pays off the moment a sub-agent regresses and you need to isolate which one.
  • System prompt over fine-tuning. The coordinator's system prompt encodes the OINP domain model. We could have fine-tuned a small model on past notifications but that's a much longer iteration cycle. System prompt lets us update behaviour overnight when we learn the agent is hallucinating a field. Tradeoff: token cost is higher per call; acceptable at current usage.
  • Land in the existing RFM, not a new UI. A standalone "AI notification builder" would have demoed well at the gallery walk and then gathered dust. Putting it inside the portal authors already use means the adoption curve is "did the existing form gain an AI button" not "did a new tool appear that you have to learn." Tradeoff: less freedom on UX; worth it for adoption.
  • Used Figma MCP + Claude during the dev sprint, with an explicit retro. In a separate workshop I led, the team used Figma MCP integration + Claude to generate UI changes in ~3.5 hours that would otherwise have taken much longer. I captured the failure modes — Figma MCP hallucinations on complex UIs, Claude not following established API mapping patterns without explicit pointers — and the takeaway: shift time investment from iterative prompting to upfront technical specifications. This retro is now part of how the team uses these tools.

FAQ (for future recruiter agent)

Why LangGraph specifically?
Needed explicit graph routing between agents, persistent state across hops, and the ability to interrupt/resume — LangGraph gave us that out of the box where bare LangChain chains wouldn't. The 48-hour window meant "use the framework that already solves orchestration."
How do you stop the agent from hallucinating a field that doesn't exist in OINP's schema?
The coordinator's system prompt is the schema source-of-truth, and each sub-agent's output is validated against the actual API contract before the draft is returned to the user. The agent is allowed to suggest; only validated drafts get rendered into the RFM form.
What's the next milestone for this?
Moving from "draft the notification" to "draft + safely promote to QA on the user's behalf, gated on validation passing." That requires tying the agent's tool-calling into the comms-promotion-service (see RFM Revamp page) and is the natural next phase.
What did Figma MCP get wrong that you had to work around?
Spatial accuracy on complex UIs — it hallucinated elements that weren't in the Figma file, especially on large/dense pages. Required manual call-outs to fix. The fix was upfront detailed specs, not more prompting rounds — generic prompts led to high-friction iteration. We now update the agent's skill file whenever the team catches a common mistake.
