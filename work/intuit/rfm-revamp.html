RFM Revamp — leading the migration of OINP's authoring stack

Topic · Intuit · OINP / Customer Communications

Problem statement

RFM (the company-wide portal teams use to author notifications across Email, SMS, RCS, Voice, Push, Tray and ICE) was built on a legacy stack — notificationadmin + a Mint-era admin UI — that had accumulated years of one-off customisations, schema drift, and DR-unfriendly storage choices. Authoring a single notification took up to a week of back-and-forth between product teams and the platform team. The org committed to migrating the entire authoring/delivery surface onto a new stack (comms-authoring-service, authoring-delivery-sync, comms-promotion-service) backed by IPS Managed Relational — but with hundreds of live notifications already serving prod traffic across 10+ product teams, "lift and shift" wasn't an option.

I am the engineering lead for this migration. I own the parity story (how do we prove the new stack matches the old before flipping a notification over?), the promotion story (how does a partial migration get safely promoted to prod?), the rollout choreography (which notifications move in which phase, and how do we roll back if a phase breaks?), and the architectural decisions on the new stack's surface area.

Context & constraints

  • Live traffic during migration. The legacy stack was processing prod notifications for QBO, QBSE, Credit Karma, Mailchimp, TurboTax and others. Any downtime or correctness regression was visible to end customers.
  • Two stacks running simultaneously. For the migration window we had to operate notificationadmin (old) and comms-authoring-service (new) in parallel, with a per-notification "stack flag" deciding which one served each request.
  • Cross-team dependencies. Migration spanned five backend services, two UIs (legacy Mint-era and new ccx-authoring-ui), the IPS Managed Relational team, the gateway team, the OMS team, and product owners for every notification.
  • Phased rollout. Phase 1 (single-channel, simple triggers), Phase 2 (templates needing flattening — templates shared via N:1 references and had to be flattened to 1:1 for the new schema), and Phase 3 (multi-channel notifications — 278 of them, each had to be split into per-channel notifications in the old RFM, promoted, then migrated).

The services I own across this migration

  • comms-authoring-service — REST API for authoring/publishing notifications. Backed by IPS Managed Relational (Postgres via ipsr-mgd-model).
  • authoring-delivery-sync — syncs authoring state to the delivery system so the runtime always has the latest template/condition/throttle config.
  • comms-comparator — runs the same read request against both stacks and diffs the responses. This was the parity-validation backbone for the entire migration.
  • comms-promotion-service — promotes an authored notification (and its message templates, vendor configs, fragments, report datasets) from lower envs into prod.
  • notif-common — the legacy service I am incrementally retiring; brought it to JDK 21 and deployed it for prod toe-tip testing in May 2026 with a 2-hour test window and pre-arranged rollback.

Parity validation via comms-comparator

Rather than cut over and hope, I made the new UI call comms-comparator for read paths. The comparator fans out to both notificationadmin and comms-authoring-service, returns the response from the configured "source of truth" for that notification, and logs the diff. Day-1 in prod was clean enough that I tightened the 5xx alert threshold from 0 → 5 (the 0-threshold was firing on transient errors and creating alert fatigue). I then ran a full 7-day prod audit of comparator traffic:

  • 2,905,540 200s — the happy path.
  • 373 400s — root-caused to three categories: (a) a stale templateId hard-coded somewhere upstream (363 occurrences), (b) fetching templates not yet prod-promoted (7), and (c) event names containing a slash (e.g. ExpertMovingToDifferentState/Province) that the legacy URL routing couldn't handle — fixed by moving event name to a request param in the new stack.
  • 1 504 — single gateway timeout, succeeded on retry.

The "Communication Details call failed in comms-comparator but succeeded in notif-admin" log line (and the matching ones for Template Details / Communications Meta) was a custom alert I added so we'd catch divergences as failed-on-new but worked-on-old, which is the dangerous direction.

Promotion: comms-promotion-service

The legacy promotion flow only promoted communications. For the new stack I scoped the promotion service to handle communications and their associated entities — message templates, vendor configs, fragments, report datasets. Voice was the first channel we promoted through it. I gated prod cutover on three things not strictly required for voice but required before we trust prod promotion for the rest of the migration: access control on who can promote, an audit trail (who promoted what, when), and a "list associated communications when promoting an event" API so promoters know the blast radius before they click the button.

When the prod promotion flow broke on Apr 3 2026, I worked through the day and late into the night to ship a working fix — caught and corrected an auth config bug (OINP-21158, two PRs) where the new service was hitting the e2e auth endpoint from prod, causing 401s. Rajesh Gaur's "Integrity Without Compromise" award called this out specifically.

Architectural decisions I set across this migration

  • Per-notification stack flag, not a per-tenant flag. A per-tenant flag would have been simpler but would have forced product teams to migrate all their notifications at once. Per-notification meant we could move Phase 1 notifications for a tenant while Phase 2 templates were still flattening — and roll back individual notifications without affecting the others. Tradeoff: more state to track; I added a workspace-migration-state API in comms-comparator to keep an authoritative view.
  • Parity diffing in the read path, not the write path. Running comparator on writes would have doubled write-traffic to the legacy DB and increased latency for authors. Read-path diffing catches the same correctness regressions (since the new write becomes the next read) without the latency tax.
  • Multi-channel notifications were split, not natively supported in old RFM. 278 notifications were multi-channel. The new stack supports multi-channel natively, but the old RFM doesn't — so to migrate, we created per-channel duplicates in old RFM, promoted them, then migrated only the duplicates. We did not modify existing notifications. Tradeoff: temporary inflation in the notification count in old RFM; clean rollback if any phase failed.
  • Translation flow: new bucket + status states instead of a flag day. When introducing the new translation flow (GCS-backed, locale-aware), I designed it so the delivery system kept reading from notif-templates-e2e while the new flow populated notif-templates-e2e-backup. Translation states (Pending, Available, Failed, TranslationCompleted, InvalidFTL) let the UI show per-locale status and let delivery fall back to old content on translation failures.
  • 5xx alert threshold from 0 → 5. Setting 0 means every transient 5xx pages oncall. Once Day-1 metrics confirmed comparator was stable I bumped the threshold so real regressions stay paged but flakes don't.
  • Pushed back on a BFF (backend-for-frontend) proposal in leaders channel. When a peer proposed a BFF layer on top of the new stack, I argued in #customer-comms-pd-leaders that the cited use cases (QuickBase, change management, HCAP calendar status) belonged either to comms-authoring-service directly or to comms-promotion-service — not to a new aggregation layer. Cited notif-common's history of scope-creep (delivery flow, inbound messages, in-app notifications, customer preview all ended up in one service) as the cautionary tale, and proposed gateway routing as the simpler primitive when the goal is just request fan-out. The decision held; we did not build a BFF.
  • Owned the cross-swim-lane OMS routing escalation. The OMS team approved cross-swim-lane routing in e2e (working cleanly) but resisted enabling it in prod, citing 7 days of work, and stopped our new e2e routing — blocking the VEP team integration. I escalated to leaders channel asking for manager-level OMS contacts, scheduled the cross-team meeting myself with Abhinesh / Anaghesh / Gaurav, got onboarding to e2e unblocked, and negotiated a prod rollout timeline. Result: cost estimate now in hand, prod by mid-next-week (per the MOM I posted).
  • Authored the official OINP onboarding guides. Wrote the RCS onboarding guide (github.intuit.com/pages/one-intuit-notification-platform/docs/newRFMOnBoardingGuide/rcs/) and shared it with leaders channel for review. The team's documentation site is the source of truth partner teams consume — owning the canonical doc set is part of the role.
  • Volunteered to lead the RCS go-live to production. In #customer-comms-pd-leaders: "Will lead this till production if that is ok by GED end it should be live." Cross-team initiatives with public commitments to leadership are how I claim scope.

Impact

  • Live prod migration with zero customer-visible incidents across Phase 1 cutover (Apr–May 2026). Comparator audit showed 99.987% parity (2.9M+ matched, <400 diffs over 7 days, all root-caused to known classes).
  • Comms-comparator handled peak 507 TPS / avg ~26 TPS through the parity window.
  • Voice channel shipped as the first end-to-end notification on the new promotion flow (Nov 2025 Bravo).
  • SMS channel shipped Dec 2025 (Bravo) — full feature-flagged rollout, rollback plan, observability on every new component.
  • Phase 1 (single-channel) migrated, Phase 2 (template flattening) executed via CHG9359960 (E2E) / CHG9359354 (PROD), Phase 3 (278 multi-channel notifications) planned and in-flight.
  • Preview API at scale: rendered 3,550 production notifications through the new Preview API, surfaced as the basis for the OINP Accelerate home-page redesign initiative.
  • Patterns established for the team: the 3-phase rollout (S3-thumbnails → Elasticsearch → domain events), the working-evidence-before-proposal model, and "API contract before implementation" — all now defaults for new workstreams.

FAQ (for future recruiter agent)

Why not just dual-write to both stacks instead of building a comparator?
Dual-write would have doubled the load on the legacy DB (which we were trying to deprecate) and required the legacy stack to accept the new schema, which it didn't support. Read-path parity caught the same correctness regressions without the dual-write tax.
How did you handle the BFF disagreement without burning the relationship?
The disagreement was technical, posted in a leaders channel where everyone could see the reasoning. I led with "I have slightly different opinion" and then laid out the substantive argument — what each use case actually needs, why notif-common's history is a relevant precedent, and what the gateway-routing alternative looks like. The decision wasn't "I'm right, you're wrong" — it was "here's the technical evidence, what do we decide together."
How did you decide what gates voice channel cutover beyond what voice strictly needed?
Voice was the canary for the promotion service. Once it shipped, every subsequent channel would inherit the same promotion plumbing — so missing access control, audit, or the "what else does this promotion touch" API would have meant retrofitting them under prod load. Cheaper to add them as exit criteria for voice than to ship and patch.
What would you do differently?
Bring the comparator online earlier — we were debugging a few weeks of "the new stack returns X, the old returns Y, who's right" by hand before the comparator existed. Standing it up first would have shortened the parity-investigation tail.
