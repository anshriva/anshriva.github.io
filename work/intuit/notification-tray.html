Notification Tray — Solr/Cassandra reliability & the migration proposal

Topic · Intuit · OINP / Customer Communications

Problem statement

The OINP Notification Tray powers the in-product inbox across Intuit surfaces (QuickBooks, Mailchimp's tray, TurboTax, and several internal product surfaces) — short-lived messages, mostly 15-day TTL, ~40 GB of live data, served from a Cassandra cluster with a Solr index for query. Tray is on the hot path for every in-product notification.

Over the course of 2024–2025 the tray developed intermittent SolrCore is not available due to init failure errors that were not visible to customers (we survived on retries) but were degrading availability margin and creating perpetual oncall noise. I owned the forensics, the workaround, and ultimately the proposal to migrate the tray off Cassandra+Solr to a managed DB.

Context & constraints

  • Tray is on the user-visible path. A failed tray write doesn't just lose a log entry — the in-product inbox is missing a message. We had to be paranoid about correctness.
  • The data is short-lived. Most tray messages have 15-day TTL. This makes the tray cheap to migrate compared to long-retention stores — old data ages out faster than the migration takes.
  • Multi-region, multi-AZ. Cassandra ran across multiple AZs in us-west-2 (east cluster was the worst-affected). Solr's index was a per-tray construct; SolrCore 'oinptray.tray_message' was the specific one failing.
  • 40 GB of live data. Not huge by infra standards — large enough to be non-trivial to migrate, small enough that it's not a multi-quarter project.

Forensics

Over multiple incident windows in late 2024 and through 2025 I narrowed the failure pattern:

  • Mixed success/failure within the same host. Logs from a single tray backend pod showed config fetched from db for tray :: iep (success) and error in get tray message, name=iep error=resource not found (failure) at the same time. So it wasn't a clean "pod is dead" failure — it was per-request flakiness.
  • Time-bounded. Errors clustered between 5 PM and 7 AM IST in the east cluster. Pattern suggested off-peak compaction or GC interference with Solr core init.
  • Cassandra retries papered over user impact. A Resilience4j cassandraRetry wrapper was succeeding on retry — so customers didn't see failures, but every retry was burning latency and the underlying issue was still there. As I called out in late Nov 2025: "this issue has been going on for a month. Even now I see retrials on tray. We are just surviving on retrials. I have not been able to run proper load test due to such errors coming up."
  • Realm-level fan-out amplified pressure. Edge cases like one test realm fanning out to 559 users in a single API call were creating mini-load-spikes that interacted badly with the Solr core init issues — caught one such spike for a perf-test realm and escalated to the upstream-product team to confirm it was a test workload, not real customer traffic.

The migration proposal

By late 2025 I argued to the customer-comms team that we should stop patching this and migrate:

"Either we fix this, or we can spend a month and migrate this 40 GB tray data to a managed DB. Tray messages are short lived (mostly 15 days of TTL) so it won't be too difficult to migrate."

The economic argument: a month of focused migration ends the entire class of Solr init failures, retries, oncall noise, and load-test blockage permanently. The "fix Solr in place" path had eaten months of cycles by then with no end in sight. Short TTL means we don't have to backfill years of history — the tray naturally drains within 15 days, so a write-to-both, read-from-old, then read-from-new cutover is feasible without complex backfill machinery.

Key decisions & tradeoffs

  • Don't trust retries as the "fix." The retry wrapper was masking the issue from customers — easy to declare the problem solved and move on. I refused to call it solved because the retries themselves were a leading indicator that compaction/init pressure would eventually exceed retry budget and start dropping messages. Tradeoff: continued oncall load while the real fix was scoped; the alternative was a future surprise outage.
  • Migrate, don't patch. Cassandra + Solr is a legitimately good combination for some workloads — but the operational cost on a short-TTL workload like tray is wildly disproportionate. A managed DB (IPS Managed Relational or similar) gives us encryption, regional DR, schema promotion, observability for free. Tradeoff: one month of engineering investment; recovered in oncall hours within a quarter.
  • Leverage short TTL for cutover simplicity. Long-retention migrations need backfill tooling, replay frameworks, dual-read consistency checks. Short TTL means we can dual-write for ~15 days and then the old store is empty. Don't build infrastructure for problems you don't have. Tradeoff: dual-write window has its own correctness risks; mitigated by tray data being non-financial.
  • Per-host forensics, not just per-cluster. Early debugging assumed "east cluster is bad." Drilling into single-host logs showed same-host mixed success/failure, which falsified the cluster-level hypothesis and pointed at intra-process state (Solr core init). The investment in narrower telemetry paid off. Tradeoff: more time in Splunk; faster convergence on the real cause.

FAQ (for future recruiter agent)

Why was the same host returning both success and failure?
Solr core init is an in-process state machine — if the core failed to initialize on a particular request path (likely racing with a compaction or restart), subsequent requests for the same tray would 404 until the core came back, but other in-flight requests using a different code path could still succeed. This is exactly the kind of "you can't tell from cluster-level metrics" failure that needs per-host log slicing.
Why not just restart the affected pods on Solr errors?
Considered. The error is per-tray-core, not per-pod — restarting the pod means dropping in-flight successful requests too. The retry wrapper was strictly better than auto-restart. The actual fix is removing Solr from the path.
What managed DB are you migrating to?
Most likely IPS Managed Relational (Aurora PostgreSQL) — paved-road choice, integrates with the rest of the new stack. The decision is being made now; the proposal is the page-relevant artifact.
What's the rollback plan if the migration goes wrong?
The dual-write window is the rollback plan — for the first ~15 days, every write goes to both stores and reads can be flipped back to the old store via a single config flip. After the TTL drains, the old store has no data, so the "rollback" becomes a fresh re-cutover rather than a revert.
