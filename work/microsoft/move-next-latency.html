Move-next latency (60×)

Topic · Microsoft · Graph Connectors

Problem statement

Crawls of large file share repositories were extremely slow. Move-next operations (iterating through data source items) took ~30 seconds per call, making crawl time prohibitive for customers like Equinor and Napier City Council. Without investigation, we assumed network latency was the bottleneck — but the real issue was hidden in memory pressure.

Context & constraints

This was one of the highest-value customer adoption blockers. Large enterprises need to crawl millions of items. At 30s per move-next, we couldn't reach crawl completion. Investigation required access to customer telemetry and the ability to profile actual workloads.

Investigation approach

  • Instrumentation: Added logs to isolate move-next time from other operations
  • Profiling: Captured Windows performance dumps and memory snapshots from customer setups
  • Root cause: Discovered excessive logging was thrashing memory, causing GC pauses during critical sections
  • Fix: Made logging conditional via flight flags, reducing noise while preserving debuggability

Key decisions & tradeoffs

  • Logging over visibility: Initially assumed network latency, but profiling revealed memory bottleneck — invested in tools before jumping to fixes
  • Flight-based logging: Rather than disable logging entirely, gated it behind flights so we can turn it on per-customer for debugging without affecting baseline performance
  • Cross-team collaboration: Paired with architect (Yogesh) to validate assumptions and unblock investigation early

Impact

Metric: Reduced move-next latency from 30 seconds to 0.5 seconds — a 60× improvement.

Customer adoption: Unblocked Equinor, Napier City Council, and FTB to reach crawl completion. These customers went from impossible timelines to viable deployments.

Scale impact: Expected to index millions of items across large enterprises once crawls became feasible.
