Admin Self-Serve visibility and tooling

Feature · Microsoft · Graph Connectors

Problem statement

Admins couldn't see what was happening to their crawls. When issues occurred, they had no visibility into error details, failure reasons, or remediation steps — forcing them to file support tickets for every problem. The team's velocity was bottlenecked by customer escalations that could have been self-resolved.

Context & constraints

This was one of the biggest operational pain points identified by leadership. A Future Hack Lounge (FHL) was organized to explore "ideal admin experience." The feature team included designers, PMs, and engineers. Success meant reducing escalations and giving admins actionable intelligence directly in the UI.

Approach

  • FHL design sprint: Hackathon-style workshop where the team mocked up the ideal admin experience — what would admins need to see and do without calling support?
  • Backend APIs: Designed new admin APIs to expose crawl state, error details, and remediation actions
  • Connector integration: Updated connector code to populate new data fields and status transitions
  • UI/UX: Built admin dashboard showing real-time crawl status, error messages, and suggested next steps

Key decisions & tradeoffs

  • Actionable errors over generic messages: Instead of "crawl failed," show specific errors like "file permissions denied" with remediation steps
  • Real-time visibility: Dashboard updates live as crawls progress, so admins can monitor without polling support channels
  • Cross-functional design: Invested in workshops with designers and PMs upfront to get the UX right before heavy backend work

Impact

Recognition: The FHL hack received special mention from judges at the company-wide hackathon and was recognized by multiple stakeholders as a breakthrough idea.

Operational shift: Expected to reduce customer escalations significantly by enabling admins to diagnose and resolve issues independently. Success metric: reduction in days needed to resolve a customer issue.

Team velocity: Freed the engineering team from low-value triage work, enabling focus on platform improvements rather than support firefighting.

Ongoing: Engineering requirements and module design in progress. Expected rollout in 2024.
