ServiceNow ACL model

Topic · Microsoft · Graph Connectors

Problem statement

ServiceNow ticket connectors lacked proper access control, causing tickets to be indexed even when users should not have access. The existing ACL model was insufficient — crawl flows didn't respect fine-grained permission boundaries, and caching wasn't optimized, leading to redundant API calls and stale data.

Context & constraints

ServiceNow is a critical enterprise data source for IT workflows. Customers expect search to enforce the same access controls as ServiceNow itself. The ACL revamp required coordinated changes across API design, crawl flow, and caching strategy.

Approach

  • API design: Redesigned admin APIs to expose and track access control boundaries per ticket
  • Crawl flow: Updated connector logic to respect ServiceNow ACLs during index time
  • Caching strategy: Implemented negative caching and ticket-level caching to reduce API load while maintaining freshness
  • Validation: Worked with multiple customers to test real-world permission scenarios before GA

Key decisions & tradeoffs

  • Negative caching: Cache ACL denials to avoid redundant "permission denied" calls — a common pattern when users search across many tickets
  • Per-ticket caching: Balance freshness (TTL per ticket) with API quota, since ServiceNow has rate limits
  • Customer validation first: Brought multiple customers into beta before GA to catch real-world access patterns we might have missed

Impact

Feature status: Implemented in Nov 2023, currently being validated by multiple customers. Expected to reach GA after customer sign-off.

Security: Ensures search respects ServiceNow's access model — users can no longer see tickets they shouldn't access through search.

Operational efficiency: Negative and ticket-level caching reduced API calls to ServiceNow by filtering out redundant requests, lowering costs and latency.
