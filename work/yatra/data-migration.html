Moving the booking database while bookings kept arriving

Topic · Yatra · Booking data platform · Senior SE · execution under an architect's design

What we were getting off of

After my promotion in 2018 I moved to the team rebuilding the booking data platform. The old store was Microsoft SQL Server, and the data in it was generated by a third-party software package. That package wrote a sprawl of tables and leaned on stored procedures thousands of lines long — complex, tool-specific queries that existed because the tool wanted them that way, not because the domain did. The company didn't want to keep paying for the SQL Server subscription, and more than that didn't want the data model held hostage to one vendor's tool. The plan was to move to in-house PostgreSQL, with Yatra's own tooling to access it.

I owned the bus and train slices of the migration — the domains I already knew from the previous team. I want to be precise about scope: an architect designed the target system. My job was to implement against that design — to add the Kafka topics, the database tables, and the new databases the architecture called for, and to get the bus and train data across correctly.

Why "just migrate it" wasn't an option

A database migration is easy if you can stop the world: freeze writes, copy everything, point the application at the new store, unfreeze. A live booking platform never gives you that window. Bookings, payments, and changes keep landing the entire time the migration runs, so any row you copy can be stale the moment after you copy it, and the new store is wrong the instant it goes live. You have to handle the historical backfill and the live stream of new writes at the same time, and you have to keep them consistent.

The change pipeline, and a read model built for speed

The shape of the solution was a pipeline that kept the new store current with the old one continuously: whenever a change happened in the old database, the same change was carried into the new database right away, so reads could always be served from the new store with confidence. In practice a booking payment completing in the old system showed up in the new one within a second or two — close enough to live that the new store was never meaningfully behind. That's what let the cutover be safe: the target wasn't a stale copy you'd have to reconcile, it was a continuously-updated mirror.

The migration also bought a performance win, because the new model wasn't a transcription of the old one. The old stored procedures assembled a single record through too many joins — the cost of a schema the tool had shaped. The new design kept the data denormalized into a client-ready form. The relational Postgres side still needs some joins, so on top of it we also stored a fully denormalized copy in MongoDB: key = booking ID, value = the complete booking as JSON. A lookup that used to fan out across joins became a single keyed read. Record retrieval went from a few seconds to a few milliseconds.

The old store keeps taking live writes; the pipeline mirrors each one into the new stores within a second or two, so reads can move to Postgres + MongoDB before the old store is retired.

What I took from it

The platform came off a licensed database and a vendor-shaped schema onto in-house Postgres with a MongoDB read model, and reads got dramatically faster as a side effect of modelling the data for how it's actually queried rather than how a tool happened to store it. My part was execution at depth — owning the bus and train data end to end and building the topics, tables, and stores the architecture called for — not the architecture itself, and I'd rather state that plainly than dress it up. The lesson that stuck is the one a live cutover teaches you the hard way: the migration isn't the copy, it's keeping two stores honest with each other while the ground keeps moving underneath both.
